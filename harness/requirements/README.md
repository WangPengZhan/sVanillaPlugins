# Requirements

Use this section for user goals, product requirements, and acceptance criteria.

Good entries include:

- feature requirements;
- compatibility requirements;
- user-visible behavior expectations;
- explicit non-goals;
- acceptance criteria that can be checked by tests or harness commands.

Requirements should say what must be true, not how the code must be written.
Implementation details belong in `../specs/` or the source tree.
